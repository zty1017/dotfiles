# AI 原生个人开发环境方案

## 1. 文档目的

这份文档用于定义一套面向以下场景的个人开发环境与工作流方案：

- 多设备、多服务器开发
- 远程优先开发
- 大量使用终端工具
- 同时使用多个 AI 编码工具
- 使用 `git worktree` 进行任务隔离和并行推进
- 使用 dotfiles 在多台机器之间保持行为一致

这份文档的用途有两个：

- 作为后续其他 AI 审核、补充、比较和 refinement 的基础上下文
- 作为本人后续迭代 dotfiles、工作流和工具链的系统设计说明

这不是一份简单的工具清单，而是一份完整的系统方案，内容包括：

- 工作流设计
- 工具分层
- 为什么选这些工具
- 各工具的职责边界
- AI agent 的协作方式
- dotfiles 的组织思路
- 未来可扩展方向

---

## 2. 用户画像与约束

当前已知使用场景如下：

- 常用 `VS Code` 进行较重的代码编写和多文件编辑
- 大量使用 `Vim` 看代码、查代码、做轻量修改
- 经常使用 `tmux`
- 经常使用 `git worktree`
- 经常使用终端型 AI 编码工具
- 已知常用 AI 工具包括：
  - `OpenCode`
  - `Gemini CLI`
  - `Codex`
  - `Claude Code`
  - 偶尔使用 `Cursor` 等 AI IDE
- 有两台实验室服务器，无 root 权限
- 有三台个人云服务器
- 有两台个人 PC
- 有一台实验室电脑

由此可推出几个非常重要的约束：

- 环境必须支持无 root 场景
- 环境必须支持远程优先
- 环境必须支持多机器行为尽量一致
- 环境不能强依赖单一 IDE
- 环境必须适配“一个任务一个 worktree 一个 AI 会话”的模式
- 环境应该优先优化“阅读、导航、轻编辑、审阅、并行任务管理”，而不是只优化“单窗口里写代码”

---

## 3. 总体结论

推荐的核心架构是：

- `Git` 负责历史、回滚、审阅和最终收敛
- `git worktree` 负责任务隔离和并行推进
- `tmux` 负责会话编排、持久化和多面板工作流
- `Vim` 负责主力代码阅读、导航和轻编辑
- `VS Code` 负责较重的代码编写、重构和调试
- 终端型 AI agent 作为主力 AI 执行层
- dotfiles 负责跨设备的一致性和可迁移性

不建议把整个开发系统建立在下面任何一种单点假设上：

- “一个 IDE 就能解决所有事情”
- “一个 AI 编码工具就应该成为唯一主力”
- “只要多装插件，终端环境自然就会好用”

更合适的思路是分层：

- 终端是主操作面
- AI agent 是可替换执行器
- Git 是最终真相来源
- worktree 是任务隔离单位
- 结构化规则文件和 skills 用于稳定 AI 行为

---

## 4. 设计目标

这套环境的目标不是“炫技”，而是尽量解决真实开发中的高频痛点。

核心目标如下：

### 4.1 保证正式产出可追踪、可回滚、可审查

AI 会参与越来越多工作，但最终产出仍然必须能回到：

- Git 历史
- diff
- 测试
- review

### 4.2 让多个任务天然隔离

一个任务应尽量具备：

- 独立目录
- 独立分支
- 独立 AI 会话
- 尽量独立的运行时输出

### 4.3 优化多 worktree 工作方式

这套环境不是为“单项目单窗口单上下文”设计的，而是为：

- 一边做 feature
- 一边修 bug
- 一边补文档
- 一边跑实验
- 一边让 AI 并行处理多个不同任务

### 4.4 优化远程开发体验

重点不是桌面花哨程度，而是：

- 远程稳定性
- 无 root 可部署
- shell 工具顺手
- 会话不断
- 日志和任务能长期挂着

### 4.5 让 AI 工具真正提高吞吐，而不是制造混乱

AI 应该帮助：

- 调研和总结
- 样板代码生成
- 局部重构
- 补测试
- 补文档
- 分析 bug
- 生成实验变体

而不是让系统变成：

- 一堆工具互相打架
- 一堆 session 无法管理
- 一堆临时文件到处乱飞

---

## 5. 核心原则

### 5.1 Git 是最终权威

所有正式产出都必须满足：

- 可被 diff
- 可被 review
- 可被回滚
- 可被合并

AI 的输出不应直接视为“可信完成”，而应被视作候选变更。

### 5.2 一个任务 = 一个分支 = 一个 worktree = 一个主 AI 会话

这是本方案最关键的稳定性约束之一。

它能显著减少：

- 上下文污染
- 分支切换成本
- AI 会话跑偏
- 多任务共用目录导致的混乱

### 5.3 终端优先，IDE 补充

这套环境不是完全排斥 IDE，而是明确分工：

- `Vim` 负责高频阅读和轻编辑
- `VS Code` 负责重编辑和图形调试
- `tmux + shell + AI CLI` 负责日常主工作流

### 5.4 AI 是并行执行器，人是裁决者

AI 更适合做：

- 高重复工作
- 信息整理
- 局部实现
- 局部修复
- 补充性文档和测试

人更适合做：

- 任务拆解
- 边界控制
- 架构判断
- 结果取舍
- 最终验收

### 5.5 轻量、可迁移、无 root 优先

由于设备条件复杂，方案应优先选择：

- 用户目录可安装的工具
- shell 友好的工具
- 对系统依赖少的工具
- 即使环境不理想也能降级使用的工具

---

## 6. 为什么 `git worktree` 应该成为结构核心

传统单目录工作流在 AI 时代的问题很明显：

- 不同任务切换需要 `stash/switch/checkout`
- AI 把当前文件树当成稳定上下文，但你可能随时切分支
- 多个 AI 会话容易相互污染
- 日志、缓存、端口和临时文件容易互撞

`git worktree` 的优势在于：

- 一个仓库共享一套历史和对象库
- 多个任务可以有多个独立工作目录
- 每个目录可以检出不同分支
- 不需要频繁 clone
- 特别适合“一个任务一个目录一个上下文”

本方案默认假设：

- 主 worktree 尽量保持稳定
- 所有有意义的 side task 尽量放到 linked worktree
- worktree 不是永久囤积的，而是有生命周期管理的

适用任务包括：

- feature
- bugfix
- docs
- refactor
- experiment
- research

## 7. 推荐的总体工作流

### 7.1 标准任务生命周期

推荐的任务流程如下：

1. 人先定义任务
2. 创建分支和 worktree
3. 初始化任务说明文件
4. 起标准 tmux 工作布局
5. 为该 worktree 分配一个主 AI 会话
6. 并行进行阅读、编码、测试、日志观察和 AI 辅助
7. 人工 review 和收敛
8. 合并或放弃
9. 删除 worktree 并清理元数据

### 7.2 标准任务文件

每个非 trivial 的 worktree 应尽量带一个 `TASK.md`，内容建议包括：

- 任务目标
- 不做什么
- 验收标准
- 重点文件
- 约束条件
- 验证步骤
- 风险点
- 未决问题

这会显著提升：

- AI 任务稳定性
- 任务交接能力
- 审阅质量
- 长期记忆能力

### 7.3 标准项目级 AI 说明

项目建议维护一个 `AGENTS.md`，说明：

- 仓库结构
- 常用命令
- 测试命令
- 代码规范
- 哪些目录可改
- 哪些目录尽量不要动
- review 要求

这个文件应作为各类 AI agent 的统一项目上下文。

### 7.4 标准知识沉淀目录

建议为项目留出轻量的知识沉淀空间，例如：

- `notes/research/`
- `notes/decisions/`
- `notes/experiments/`

这类内容不适合全部塞进 README，但也不应全部丢失。

## 8. 工具分层设计

## 8.1 版本控制层

### 工具

- `Git`
- `git worktree`

### 为什么需要这层

这层负责：

- 版本控制
- 差异审查
- 分支隔离
- 最终合并
- 回滚

### 使用方式

- 主线尽量稳定
- 任务尽量小而明确
- 不同任务不要共用一个工作目录
- 任务完成后进行清理

### 为什么选这些工具

这仍然是软件开发最稳、最通用、最具兼容性的基础设施。即使未来 AI 工具持续变化，Git 层也仍然是最稳定的收敛面。

---

## 8.2 会话与工作台层

### 工具

- `tmux`

### 为什么需要这层

这层负责：

- 远程会话持久化
- 多 pane 协作
- 测试、日志、AI、编辑同时在线
- 长时间任务不中断

### 推荐使用方式

默认可采用两种模型：

- 一个 worktree 对应一个 tmux window
- 一个 worktree 对应一个 tmux session

推荐标准 pane 布局：

- pane 1：`vim`
- pane 2：主 AI CLI 工具
- pane 3：测试或 shell
- pane 4：日志或 `git diff`

### 为什么选 `tmux`

- 远程环境友好
- 无 root 成本低
- 生态成熟
- 已与当前工作流高度匹配

---

## 8.3 阅读与轻编辑层

### 工具

- `Vim`
- `ripgrep`
- `fzf.vim`
- `NERDTree`
- `fd`
- `bat`

### 为什么需要这层

当前工作流中，最常发生的动作不是“深度写代码”，而是：

- 看代码
- 跳转代码
- 搜文件
- 搜字符串
- 改一两处
- 快速验证

### `Vim` 的定位

在本方案中，`Vim` 应定义为：

- 主代码阅读器
- 主导航器
- 主轻编辑器
- 远程环境默认编辑器

它不是“偶尔应急工具”，而是主工作面的一部分。

### 推荐的轻量插件增强

推荐补充：

- `vim-commentary`
- `vim-surround`

原因：

- 高频
- 低维护
- 低心智负担
- 对轻编辑和代码微调非常有价值

### 为什么不主推重型 Vim 生态

并不是说 LSP/Treesitter/DAP 没价值，而是当前主矛盾不在“把 Vim 变成完整 IDE”，而在：

- 多上下文切换
- 阅读效率
- 审阅效率
- worktree 管理
- AI 协作稳定性

---

## 8.4 重编辑层

### 工具

- `VS Code`
- 按需使用 `Cursor` 等 AI IDE

### 为什么需要这层

有些任务依然更适合图形 IDE：

- 大规模多文件修改
- 结构化搜索替换
- 调试
- 较重代码编写
- 需要大量上下文联动时

### `VS Code` 的定位

`VS Code` 不应作为多任务总调度器，而应作为：

- 重编辑工具
- 深度编写工具
- 图形调试工具

它是补充层，而不是唯一主操作面。

---

## 8.5 AI 执行层

### 当前主用工具

- `OpenCode`
- `Gemini CLI`
- `Codex`
- `Claude Code`

### 次级或按需工具

- `Aider`
- `GitHub Copilot CLI`
- `Cursor Background Agents`
- `GitHub Copilot coding agent`

### 为什么保留多个 AI 工具

当前没有任何一个 AI 编码工具在以下所有维度都全面领先：

- 开源性
- 免费性
- 终端体验
- 模型质量
- 背景执行能力
- IDE 集成
- 插件/MCP/skills 生态

因此，更合理的做法是：

- 保留一个默认主力本地 agent
- 保留一到两个高质量备选 agent
- 按需使用异步后台 agent

## 9. 各类 AI 编码工具的角色定义

## 9.1 OpenCode

### 建议定位

默认主力本地终端 agent。

### 为什么选它做主力

- 开源
- 免费
- 终端优先
- 支持多 session
- 支持 MCP
- 与当前终端工作流高度一致

### 最适合做什么

- 日常主力 coding agent
- 多 worktree 并行任务
- 文档、测试、重构、局部分析

### 风险

- 和其他 agent 一样，容易生成过量中间文件
- MCP 接太多容易增加上下文噪声

---

## 9.2 Gemini CLI

### 建议定位

高价值的第二本地 agent 和模型路线补充。

### 为什么保留

- 提供模型多样性
- 终端 agent 能力成熟
- 支持 MCP
- 适合做交叉验证

### 最适合做什么

- 第二意见
- 分析型任务
- 与其他 agent 做对照

---

## 9.3 Codex

### 建议定位

高质量 coding/review 路线。

### 为什么保留

- 在重要实现、深度推理、代码审阅中可作为高质量补充
- 适合在关键任务上做二次验证

### 最适合做什么

- 复杂实现
- 深度审阅
- 较高风险任务的复核

---

## 9.4 Claude Code

### 建议定位

面向长链路、复杂 agent 工作流的强力终端 agent。

### 为什么保留

- 更适合多步骤、长时、工具密集型任务
- 在一些复杂任务上可以作为主力备选

### 最适合做什么

- bug 深挖
- 长流程调研
- 工具链较复杂的任务

---

## 9.5 Aider

### 建议定位

Git 工作流很顺的辅助型 agent。

### 为什么保留

- git 集成好
- 小范围变更体验好
- 适合有控制感的迭代修改

### 最适合做什么

- 小步修改
- 局部重构
- 以 git diff 为中心的小范围任务

---

## 9.6 后台异步 agent

### 工具

- `Cursor Background Agents`
- `GitHub Copilot coding agent`

### 建议定位

异步外包层。

### 最适合做什么

- 文档清理
- 小型 bugfix
- 补测试
- 低风险技术债处理

### 为什么不是主力

当前主工作流是：

- 终端优先
- worktree 优先
- 本地可控优先

因此它们更适合作为补充，而不是主骨架。

## 10. MCP、插件、Skills 是否应该建设

答案是：应该，但要克制地建设。

---

## 10.1 总体判断

现代 agent 越来越依赖：

- MCP
- plugins/extensions
- skills

但本方案不建议走“全都装上”的路线，而建议走：

- 少量高价值 MCP
- 稳定、可复用的 skills
- 少量真正高频的插件

目标不是让工具数量最大化，而是让：

- 重复工作标准化
- 上下文更稳定
- 跨 agent 复用更强

---

## 10.2 Skills 的优先级应高于大量 MCP

原因是：

- MCP 解决“agent 能接入什么能力”
- skills 解决“agent 应该如何稳定做事”

对当前工作流来说，后者更重要。

推荐优先建设的 skills：

- `worktree-bootstrap`
- `bug-investigation`
- `test-backfill`
- `research-summary`
- `experiment-variant`
- `docs-polish`

这类 skill 应该编码：

- 任务步骤
- 输出结构
- 验证要求
- 边界条件

---

## 10.3 推荐第一批 MCP 类型

建议第一批只保留少量高价值种类：

- 文档/搜索类 MCP
- Git 或 GitHub 类 MCP
- 文件系统类 MCP
- 数据库类 MCP（仅在确实常用时）

不建议一开始就接一大堆低频 MCP，例如：

- 邮件
- 聊天工具
- 设计工具
- 低频 observability 工具

除非这些已经是日常工作流核心组成部分。

---

## 10.4 为什么要节制

MCP 和插件过多会导致：

- 上下文噪声上升
- agent 乱选工具
- 调试难度增加
- 维护成本上升

本方案更推荐：

- 工具少而稳
- 规则清楚
- 复用性高

---

## 11. 运行时隔离：不仅要隔离代码，还要隔离输出

`worktree` 只解决代码目录隔离，不自动解决运行时隔离。

必须制度化约定这些东西：

- 输出目录
- 日志目录
- 缓存目录
- 端口
- 数据库文件
- 容器名

建议默认规则：

- `outputs/<task>/`
- `logs/<task>/`
- `.cache/<task>/`
- 任务名前缀化的端口和进程命名

这对于以下场景尤其重要：

- 实验
- web 服务
- notebook
- 训练任务
- 长时间后台进程

---

## 12. 项目内建议固定的 AI 相关文件

每个值得长期维护的项目，建议逐步标准化以下文件：

- `AGENTS.md`
- `TASK.md`
- `notes/`

未来可以考虑再补：

- 项目级 `skills/`
- bug 或 experiment 模板

这些文件不是装饰性的，它们是 AI 协作控制面的组成部分。

## 13. dotfiles 策略

当前仓库可以作为 bootstrap 起点，但中期应该逐步从“单脚本安装”演化为“结构化 dotfiles 系统”。

### 建议的分层

#### 核心层

几乎所有机器共享：

- shell 配置
- git 配置
- tmux 配置
- vim 配置
- 常用 CLI 工具
- AI wrapper 和 alias
- worktree helper scripts

#### 服务器层

面向云服务器和实验室服务器：

- rootless 安装逻辑
- 更轻的工具子集
- 终端优先
- 远程安全默认值

#### 桌面层

面向本地 PC 和实验室电脑：

- `VS Code` 配置
- GUI 相关辅助
- 剪贴板集成
- 可选 IDE 增强

#### 私有层

不进入公开仓库：

- token
- host alias
- 私有环境变量
- 实验室内网配置

以下是结合当前行业生态，对这份工作流方案的深度剖析与补充扩展：

---

### 1. 任务与环境隔离：`git worktree` vs 行业趋势 (CDEs / 容器化)

文档中大量篇幅强调使用 `git worktree` 来解决“多任务并行”导致的冲突。这在纯文件层面是完美的，但行业内对隔离的追求已经到了**“运行时级别”**。

*   **当前方案的盲区（运行时冲突）：** 文档中 11 节提到了端口、数据库、缓存的冲突，并试图用“命名规范”来约束。在实际高频 AI 开发中，AI 经常会随意启停服务、修改全局环境变量（如 Python 的 pip 包，Node 的全局版本），这种基于规范的约束很容易崩溃。
*   **行业的解法：**
    *   **CDE (Cloud Development Environments)：** 像 GitHub Codespaces, Gitpod, 或者开源的 Daytona。业界主流做法是“**一个任务 = 一个容器 (Devcontainer)**”，而不是仅仅一个文件目录。
    *   **Nix / Devbox：** 即使在纯本地终端，前沿开发者也在大量使用 Nix 或基于 Nix 的 Devbox。它能保证每个 `worktree` 目录不仅代码隔离，连依赖的 Node、Python 版本、底层 C 库甚至是临时数据库都是绝对隔离的。离开目录，环境自动销毁。
*   **💡 升级建议：** 你的方案可以保留 `worktree` 的极速切换优势，但建议在每个 task 中引入类似 `direnv + Nix`（或轻量级的局部虚拟环境管理），让 AI Agent 运行在一个“即使把它搞坏了也能一键重建”的沙箱中。

### 2. 上下文管理：`TASK.md` 静态文本 vs 动态上下文构建 (RAG / AST)

文档提出用 `AGENTS.md` 和 `TASK.md` 来管理 AI 的上下文。这是非常有价值的工程实践，但现代 AI 工具已经在追求**更自动化的上下文组装**。

*   **当前方案的局限：** 纯靠人手写 `TASK.md` 成本很高。且当项目变大时，把整个项目的规范写进 `AGENTS.md`，很多 AI 会因为 Context Window 变长而“注意力涣散”。
*   **行业的解法：**
    *   **`.cursorrules` 标准化：** 行业内已经形成事实标准的项目规则文件是 `.cursorrules`（Windsurf 甚至 Copilot 也在兼容或类似概念）。建议将 `AGENTS.md` 的内容重构为符合该规范的格式，这样即使你偶尔切换到图形化 AI IDE，规则依然生效。
    *   **Codebase Indexing 工具：** 像 `repomix` (前身是 repopack) 或 `gitingest`。业界流行在任务开始前，运行一个脚本，把当前相关的 AST (语法树)、Git 历史、关联 issue 自动打包成一个高度优化的 Markdown 送给大模型。
    *   **Aider 的 Map 机制：** 像 Aider 这样的顶尖终端工具，内部使用了 Tree-sitter 生成全仓库的 Repository Map。你不需要手动告诉它仓库结构，它能自己看懂。
*   **💡 升级建议：** 将上下文分为“显式意图”（人写的 `TASK.md`）和“隐式结构”（由工具如 `repomix` 自动生成的当前目录结构和关键接口声明），两者结合喂给 Agent。

### 3. 编辑与审查：终端审阅的认知负担 vs GUI 的直觉优势

文档将 Vim 定义为“主操作面”，将 VS Code 降级为“重编辑层”。这在人类手写代码时代很合理，但在 AI 时代可能会成为瓶颈。

*   **当前方案的痛点：** AI 时代，人的核心工作从“写”变成了“审阅（Review）”。AI 经常一次性生成跨越 5 个文件、几百行的变更。在终端里用 `git diff` 或 `delta` 审阅大规模非结构化重构，认知负担极大。
*   **行业的解法：**
    *   **AI-Native IDEs 的多文件 Diff (Composer/Cascade)：** Cursor 的 Composer 和 Windsurf 的 Cascade 最强大的不是写代码，而是以 GUI 的形式展示多文件 Diff，允许人类以“文件区块”为单位进行 Accept / Reject。
    *   **PR-based Workflow (SWE-agent 模式)：** 像 Devin 或 GitHub Copilot Workspace，其核心理念是把复杂的 AI 任务隔离在后台，最终生成一个 Pull Request。人类在浏览器里（拥有最棒的 Diff 界面和评论工具）进行 Review。
*   **💡 升级建议：** 即使主力是终端，对于 AI 生成的大量代码，建议增加一个快捷命令（如 `wt-review`），该命令自动打开 VS Code 的 Source Control 视图或在本地生成一个轻量级的 Web Diff 页面，专门用于验收 AI 的产出。

### 4. 代理协作模式：单 Agent 串行 vs 多 Agent 协同流 (Multi-Agent Workflows)

文档中提到为一个 worktree 分配“一个主 AI 会话”。这属于早期的 Copilot 模式。

*   **行业的解法：** 现代 AI 框架（如 AutoGen, LangGraph, 或 Aider 的 Architect/Editor 模式）倾向于将任务拆分给不同角色的 Agent。
    *   **架构师 Agent (Architect)：** 只读，负责分析代码，提出修改方案（不写具体代码）。
    *   **工程师 Agent (Editor)：** 拿到架构师的方案，只负责精准修改文件。
    *   **测试 Agent (Tester)：** 自动跑测试，如果报错，把报错信息喂回给工程师。
*   **💡 升级建议：** 在你的 `tmux` 面板布局中，可以将单一的“主 AI 会话”升级为“AI 工作流管道”。例如，先用一个模型（如 Gemini 1.5 Pro，擅长大上下文）做规划，输出到 `PLAN.md`，然后用另一个模型（如 Claude 3.5 Sonnet 或本地的代码特化模型）执行 `PLAN.md` 里的步骤。

### 5. 工具链的冗余与降维

文档列出了长长的工具清单（Vim, VS Code, OpenCode, Gemini CLI, Codex, Claude Code, Aider, Cursor 等）。

*   **行业的反思：** 太多 AI 工具会造成严重的“心智碎片化”。每个工具的快捷键、系统提示词格式、上下文管理方式都不同。
*   **💡 升级建议（做减法）：**
    其实你只需要三个层级的工具：
    1.  **宏观推理与重构：** 一个具备超长上下文、支持多文件编辑的工具（如 **Aider** 在终端，或者 **Cursor/Windsurf** 在 GUI）。
    2.  **局部行级补全：** 一个轻量级的后台补全插件（如 GitHub Copilot 或 Supermaven）。
    3.  **万能脚本/胶水工具：** 像 **Gemini CLI** 或通过 MCP 编写的自定义 Python 脚本，专门用于“查文档、总结日志、写 Commit、分析 Bug”。不要让所有的工具都去干写代码的活。

### 总结：对这份文档的重构建议（Next-Gen Workflow）

这份文档的出发点极佳，但如果要迈向真正的 **Next-Gen AI Workflow**，建议在系统设计上做如下微调：

1.  **隔离从“文件级”上升到“环境级”：** 拥抱 `git worktree` + `Nix/direnv`，确保你的数据库和依赖不会在多个 AI 会话中打架。
2.  **上下文从“全手写”走向“半自动化”：** 保留 `TASK.md` 表达意图，引入 `repomix` 自动打包工程上下文，替换掉繁重的全局 `AGENTS.md`（或转为 `.cursorrules`）。
3.  **操作面明确“执行在终端，审查在 GUI”：** 让 AI 在 `tmux` 和终端里疯狂输出，但在最终合并（Git 审阅）时，毫无包袱地调出 VS Code 或专属 Diff 工具来降低人类的审查疲劳。
4.  **将经验固化为 MCP/Skills，而不是单纯的 Alias：** 文档中提到了很多 shell alias。在 AI 时代，应该把这些脚本包装成 **MCP Server**。这样，无论是 Gemini CLI、Claude 还是 OpenCode，都能直接调用你的 `wt-new` 或 `task-clean` 工具，让 AI 自己管理 Worktree。

结合“低成本多工具”的现实约束，我们再来重新审视和升级你的《AI 原生个人开发环境方案》。在这种约束下，**系统设计的核心难题变成了：如何在多个不同的 AI CLI 之间无缝切换，而不丢失上下文？**

以下是专门针对“出于成本考量的多 AI 工具工作流”的深度优化建议：

### 1. 确立“基于成本与能力的路由”规则 (Cost-Driven Model Routing)

既然有多个工具，就不能让它们随意混用，而应该根据任务的 **Token 消耗量** 和 **智商需求** 进行明确的“派单”。

你可以在你的 `AGENTS.md` 或个人的脑海中建立这样一个路由表：

*   **重度上下文注入 & 基础调研（免费/低成本的大胃王）：使用 Gemini CLI**
    *   **优势：** Gemini 1.5 Pro 有 100万-200万的超大上下文，且在 AI Studio 免费额度很高。
    *   **适用场景：** 丢入几万行的日志找 Bug、把整个框架的官方英文文档扔进去让它总结、初期的项目架构评估。
*   **高难度代码生成 & 复杂逻辑重构（高成本的尖刀）：使用 Claude Code 或特定高智商 API 工具**
    *   **优势：** Claude 3.5 Sonnet 目前在编码领域处于统治地位。
    *   **适用场景：** 核心算法实现、极其复杂的正则编写、多文件依赖的重构。因为贵，所以只在关键节点用，用完即走。
*   **日常 CRUD、写注释、补测试（廉价劳动力）：使用 OpenCode (配合 DeepSeek/开源模型) 等**
    *   **优势：** API 极其便宜（如 DeepSeek V3/R1）甚至本地运行免费。
    *   **适用场景：** 顺手写个 README、生成一段样板代码、写一些无聊的单元测试。

### 2. 打造“绝对可移植的上下文” (Portable Context)

这是多工具交替使用的**命门**。

当你用 Gemini 调研完，准备让 Claude 去写代码时，由于它们是两个不同的 CLI，Claude 并没有 Gemini 的聊天记录。如果你每次都靠手打描述，效率极低。

*   **原方案的闪光点：** 你文档中提出的 `TASK.md` 极其关键。这就是你的“可移植上下文”。
*   **进阶改造：标准化握手协议。**
    你需要强制所有的 AI 会话都遵循一个范式：**不依赖聊天记录，只依赖文件状态。**
    1.  当你在 `worktree-feature-A` 中，发现自己当前使用的 AI (比如 OpenCode) 搞不定了，或者快超出 API 额度了。
    2.  你立刻要求当前 AI：“把我们目前讨论的结论、未解决的问题、以及接下来要修改的文件列表，更新到 `TASK.md` 中。”
    3.  关闭 OpenCode，打开 Gemini CLI。
    4.  对 Gemini CLI 说的第一句话永远是：“**读取 `TASK.md` 和 `git diff`，告诉我你理解了当前进度，并继续执行下一步。**”

这种模式下，**文件系统（`TASK.md` 和 Git 状态）就是多 AI 协作的“共享内存”**。

### 3. 构建“多模型友好的胶水脚本”

为了降低在不同 CLI 间切换的摩擦力，你的 Dotfiles 里需要一些专门为“多 AI 协作”设计的脚本。比如，我们可以设计一个名为 `wt-context` (Worktree Context) 的小工具。

**它的作用是：一键打包当前上下文，方便你粘贴或重定向给任何最便宜/最合适的 AI。**

例如，你可以写一个简单的 bash 脚本 `wt-context.sh`：
```bash
#!/bin/bash
# 自动收集当前 worktree 的核心状态
echo "=== 任务描述 (TASK.md) ==="
cat TASK.md 2>/dev/null || echo "(无)"
echo -e "\n=== 当前未提交的修改 (git diff) ==="
git diff
echo -e "\n=== 目录结构 (tree) ==="
tree -L 2 -I 'node_modules|.git|__pycache__'
```

当你切换 AI 工具时，只需：
`wt-context > current_state.txt`
然后把 `current_state.txt` 喂给新的 AI CLI，它就能瞬间接管任务。

### 4. 彻底的“Git 驱动”：防范 AI 产生的烂尾工程

多个 AI 工具交替修改同一个 worktree，极其容易产生“脏代码叠加”。A 工具改了一半，B 工具基于 A 的错误代码继续改，最后彻底跑偏。

*   **原则：微型提交 (Micro-commits)。**
    一旦某个 AI（比如便宜的 OpenCode）完成了一部分可用代码，甚至只是搭了个架子，**立刻 `git add` 并 `git commit -m "WIP: agent draft"`**。
*   **价值：** 这样当切换到高成本 AI (比如 Claude Code) 介入时，如果它把代码改炸了，你可以毫无心理负担地 `git reset --hard`，回到上一个 AI 留下的存档点。**Git 历史成为了多 Agent 协作的 Save/Load 系统。**

### 总结

因为“成本约束”和“放弃 Aider”，你的系统反而变得更加务实：
它不再是一个追求“单一完美 Agent 自动搞定一切”的乌托邦，而是一个**由 Git Worktree 提供物理隔离、由 `TASK.md` 提供共享内存、由你本人担任调度员（路由器），根据性价比随时切换不同 AI 兵种的“特种作战指挥所”。**
