# AI 原生个人开发环境方案

## 1. 文档目的

这份文档用于定义一套面向以下场景的个人开发环境与工作流方案：

- 多设备、多服务器开发
- 远程优先开发
- 大量使用终端工具
- 同时使用多个 AI 编码工具
- 使用 `git worktree` 进行任务隔离和并行推进
- 使用 dotfiles 在多台机器之间保持行为一致

这份文档的用途有两个：

- 作为后续其他 AI 审核、补充、比较和 refinement 的基础上下文
- 作为本人后续迭代 dotfiles、工作流和工具链的系统设计说明

这不是一份简单的工具清单，而是一份完整的系统方案，内容包括：

- 工作流设计
- 工具分层
- 为什么选这些工具
- 各工具的职责边界
- AI agent 的协作方式
- dotfiles 的组织思路
- 未来可扩展方向

---

## 2. 用户画像与约束

当前已知使用场景如下：

- 常用 `VS Code` 进行较重的代码编写和多文件编辑
- 大量使用 `Vim` 看代码、查代码、做轻量修改
- 经常使用 `tmux`
- 经常使用 `git worktree`
- 经常使用终端型 AI 编码工具
- 已知常用 AI 工具包括：
  - `OpenCode`
  - `Gemini CLI`
  - `Codex`
  - `Claude Code`
  - 偶尔使用 `Cursor` 等 AI IDE
- 有两台实验室服务器，无 root 权限
- 有三台个人云服务器
- 有两台个人 PC
- 有一台实验室电脑

由此可推出几个非常重要的约束：

- 环境必须支持无 root 场景
- 环境必须支持远程优先
- 环境必须支持多机器行为尽量一致
- 环境不能强依赖单一 IDE
- 环境必须适配“一个任务一个主执行上下文”的模式
- 环境必须兼容工具内 subagent 编排和多工具并行探索
- 环境应该优先优化“阅读、导航、轻编辑、审阅、并行任务管理”，而不是只优化“单窗口里写代码”

---

## 3. 总体结论

推荐的核心架构是：

- `Git` 负责历史、回滚、审阅和最终收敛
- `git worktree` 负责任务隔离和并行推进
- `tmux` 负责会话编排、持久化和多面板工作流
- `Vim` 负责主力代码阅读、导航和轻编辑
- `VS Code` 负责较重的代码编写、重构和调试
- 终端型 AI agent 作为主力 AI 执行层
- dotfiles 负责跨设备的一致性和可迁移性

不建议把整个开发系统建立在下面任何一种单点假设上：

- “一个 IDE 就能解决所有事情”
- “一个 AI 编码工具就应该成为唯一主力”
- “只要多装插件，终端环境自然就会好用”

更合适的思路是分层：

- 终端是主操作面
- AI agent 是可替换执行器
- AI 工具内建的 subagent 能力应被视为正式能力，而不是例外
- Git 是最终真相来源
- worktree 是任务隔离单位
- 结构化规则文件和 skills 用于稳定 AI 行为

## 4. 设计目标

这套环境的目标不是“炫技”，而是尽量解决真实开发中的高频痛点。

核心目标如下：

### 4.1 保证正式产出可追踪、可回滚、可审查

AI 会参与越来越多工作，但最终产出仍然必须能回到：

- Git 历史
- diff
- 测试
- review

### 4.2 让多个任务天然隔离

一个任务应尽量具备：

- 独立目录
- 独立分支
- 独立 AI 会话
- 尽量独立的运行时输出

### 4.3 优化多 worktree 工作方式

这套环境不是为“单项目单窗口单上下文”设计的，而是为：

- 一边做 feature
- 一边修 bug
- 一边补文档
- 一边跑实验
- 一边让 AI 并行处理多个不同任务

### 4.4 优化远程开发体验

重点不是桌面花哨程度，而是：

- 远程稳定性
- 无 root 可部署
- shell 工具顺手
- 会话不断
- 日志和任务能长期挂着

### 4.5 让 AI 工具真正提高吞吐，而不是制造混乱

AI 应该帮助：

- 调研和总结
- 样板代码生成
- 局部重构
- 补测试
- 补文档
- 分析 bug
- 生成实验变体

而不是让系统变成：

- 一堆工具互相打架
- 一堆 session 无法管理
- 一堆临时文件到处乱飞

## 5. 核心原则

### 5.1 Git 是最终权威

所有正式产出都必须满足：

- 可被 diff
- 可被 review
- 可被回滚
- 可被合并

AI 的输出不应直接视为“可信完成”，而应被视作候选变更。

### 5.2 一个任务 = 一个分支 = 一个 worktree = 一个主执行上下文

这是本方案最关键的稳定性约束之一。

它能显著减少：

- 上下文污染
- 分支切换成本
- AI 会话跑偏
- 多任务共用目录导致的混乱

这里的“主执行上下文”不等于“只能有一个 AI”。

更准确的约束是：

- 一个任务应有一个主写入面
- 可以存在多个辅助分析面
- 可以存在工具内部的 subagent 编排
- 最终代码改动应收敛到单一执行面，避免多个外部 agent 无约束同时写同一份代码

### 5.3 终端优先，IDE 补充

这套环境不是完全排斥 IDE，而是明确分工：

- `Vim` 负责高频阅读和轻编辑
- `VS Code` 负责重编辑和图形调试
- `tmux + shell + AI CLI` 负责日常主工作流

### 5.4 AI 是并行执行器，人是裁决者

AI 更适合做：

- 高重复工作
- 信息整理
- 局部实现
- 局部修复
- 补充性文档和测试

人更适合做：

- 任务拆解
- 边界控制
- 架构判断
- 结果取舍
- 最终验收

### 5.5 轻量、可迁移、无 root 优先

由于设备条件复杂，方案应优先选择：

- 用户目录可安装的工具
- shell 友好的工具
- 对系统依赖少的工具
- 即使环境不理想也能降级使用的工具

### 5.6 共享规则优先，工具适配其次

现代 AI 编码工具普遍支持各自的自定义指令、rules、memory、project instructions。

因此不应把规则只写成某个单一工具专属格式，而应分层：

- 共享规则层：项目结构、命令、边界、验收要求
- 个人偏好层：输出风格、个人习惯、常用流程
- 工具适配层：映射到 `AGENTS.md`、`CLAUDE.md`、Cursor Rules、OpenCode rules 等不同格式

这样做的好处是：

- 降低工具切换成本
- 提高跨工具复用度
- 避免每换一个 agent 就重写一套规则
- 让 dotfiles 中的 AI 配置更容易独立拆分和公开复用

## 6. AI 辅助编码方法论

除了工具本身，AI 时代还出现了许多“如何组织编码过程”的方法论。

本方案认为，不应把这些方法当成互斥的宗教，而应把它们视为不同任务类型下的执行策略。

更准确的做法是：

- 用合适的方法论描述任务
- 用合适的规则文件承载约束
- 用合适的 agent 执行
- 最终统一回到 review 和 Git 收敛

### 6.1 Spec-Driven Coding

即规格驱动开发。

核心思想是：

- 先定义目标
- 先定义边界
- 先定义验收标准
- 然后再让 AI 或人去实现

在当前工作流里，对应的载体通常是：

- `TASK.md`
- 必要时的 `PLAN.md`

适合场景：

- 新 feature
- 中大型重构
- 跨文件修改
- 风险较高的任务
- 需要多人或多 agent 接力的任务

优点：

- 可以降低 AI 跑偏概率
- 有利于多 agent 接力
- 更适合复杂任务收敛

局限：

- 前期写规格会增加一些成本
- 对极小任务可能显得过重

本方案建议：

- 默认把 `TASK.md` 视为最小规格文件
- 非 trivial 任务优先采用 spec-driven 思路

### 6.2 Plan-Driven Coding

即计划驱动开发，也可以理解为 architect-then-implement。

核心思想是：

- 先规划步骤
- 再分阶段执行
- 必要时由 planner 和 executor 分工

在当前工作流里，对应载体通常是：

- `PLAN.md`
- 或 `TASK.md` 中的步骤分解区块

适合场景：

- 大任务
- 多文件耦合任务
- 需要工具内 subagent 编排的任务
- 需要多模型并行分析后收敛的任务

优点：

- 适合复杂任务拆解
- 适合 planner / executor / reviewer 分工
- 适合并行探索后再统一执行

局限：

- 计划本身可能过度设计
- 如果任务太小，收益不明显

本方案建议：

- 对复杂任务优先先出 `PLAN.md`
- 工具内支持 subagent 的场景，可让 plan agent 先做分析，再由执行面落地

### 6.3 Test-Driven Coding

即测试驱动开发。

核心思想是：

- 先定义行为验证
- 再写实现
- 或至少在改实现之前先明确测试方式

对当前工作流来说，不建议把 TDD 机械化地套到所有任务上，但它对某些任务非常有价值。

适合场景：

- bugfix
- 高风险核心逻辑
- 回归风险高的模块
- AI 容易“看起来对，实际上错”的任务

优点：

- 能显著提高可验证性
- 降低 agent 幻觉式成功
- 更适合 bugfix 收敛

局限：

- 对探索性代码未必高效
- 对文档和调研类任务不适用

本方案建议：

- 对 bugfix 和高风险逻辑优先采用测试先行或测试同步补齐
- 对探索性任务可以先验证思路，再回补测试

### 6.4 Review-Driven Coding

即审查驱动开发。

这是本方案中最重要、最普遍适用的总原则。

核心思想是：

- 不假设 AI 输出天然可信
- 把 diff、测试、日志、代码阅读作为主收敛面
- 人工 review 不是收尾动作，而是主流程的一部分

适合场景：

- 几乎所有 AI 参与任务

优点：

- 与 Git 天然兼容
- 适合多 agent 协作
- 适合长期维护

局限：

- 对人的审阅能力要求较高
- 如果没有好的 diff、日志和上下文工具，容易疲劳

本方案建议：

- 默认把所有 AI 辅助任务都纳入 review-driven 框架
- 其他方法论都应最终回到 review-driven 收敛

### 6.5 四种方法如何组合

这些方法不是互斥关系。

推荐的组合方式是：

- 默认总原则：`Review-Driven`
- 新 feature 和复杂任务：`Spec-Driven`
- 大任务和多 agent 协作：`Plan-Driven`
- bugfix 和高风险逻辑：`Test-Driven`

也就是说，比较合理的顺序通常是：

1. 先用 spec 定义边界
2. 必要时用 plan 拆解步骤
3. 对关键行为用 test 锁定
4. 最后用 review 收敛结果

### 6.6 方法论与文件载体的映射

建议建立如下映射关系：

- `TASK.md`
  主要承载 spec-driven 的任务目标、边界和验收标准
- `PLAN.md`
  主要承载 plan-driven 的步骤拆解和执行计划
- `tests/`
  主要承载 test-driven 的行为验证
- `git diff`、`git show`、测试结果、日志
  主要承载 review-driven 的收敛过程

这意味着：

- 不同方法论不是抽象口号
- 它们应当在工作树中有明确载体
- AI 与人都能围绕这些载体协作

## 7. 为什么 `git worktree` 应该成为结构核心

传统单目录工作流在 AI 时代的问题很明显：

- 不同任务切换需要 `stash/switch/checkout`
- AI 把当前文件树当成稳定上下文，但你可能随时切分支
- 多个 AI 会话容易相互污染
- 日志、缓存、端口和临时文件容易互撞

`git worktree` 的优势在于：

- 一个仓库共享一套历史和对象库
- 多个任务可以有多个独立工作目录
- 每个目录可以检出不同分支
- 不需要频繁 clone
- 特别适合“一个任务一个目录一个上下文”

本方案默认假设：

- 主 worktree 尽量保持稳定
- 所有有意义的 side task 尽量放到 linked worktree
- worktree 不是永久囤积的，而是有生命周期管理的

适用任务包括：

- feature
- bugfix
- docs
- refactor
- experiment
- research

## 8. 推荐的总体工作流

### 8.1 标准任务生命周期

推荐的任务流程如下：

1. 人先定义任务
2. 创建分支和 worktree
3. 初始化任务说明文件
4. 起标准 tmux 工作布局
5. 为该 worktree 确定一个主执行面
6. 并行进行阅读、编码、测试、日志观察和 AI 辅助
7. 人工 review 和收敛
8. 合并或放弃
9. 删除 worktree 并清理元数据

### 8.2 标准任务文件

每个非 trivial 的 worktree 应尽量带一个 `TASK.md`，内容建议包括：

- 任务目标
- 不做什么
- 验收标准
- 重点文件
- 约束条件
- 验证步骤
- 风险点
- 未决问题

这会显著提升：

- AI 任务稳定性
- 任务交接能力
- 审阅质量
- 长期记忆能力

### 8.3 标准项目级 AI 说明

项目建议维护一份共享规则，再按工具做适配。

最小可行做法是保留一个 `AGENTS.md`，说明：

- 仓库结构
- 常用命令
- 测试命令
- 代码规范
- 哪些目录可改
- 哪些目录尽量不要动
- review 要求

更完整的做法是：

- 共享规则保留在 `AGENTS.md` 或等价文件中
- 对支持项目记忆/规则的工具再生成适配层
- 例如 `CLAUDE.md`、Cursor Rules、OpenCode 的项目规则文件

目标不是维护许多互相漂移的规则副本，而是：

- 共享规则写一次
- 工具按需映射

### 8.4 可移植上下文与任务接力

多工具并行和切换的关键，不是聊天记录，而是可移植上下文。

推荐把以下内容视为跨 agent 的共享内存：

- `TASK.md`
- `AGENTS.md`
- `git diff`
- `notes/`
- 必要时的 `PLAN.md`

当需要从一个 AI 工具切到另一个工具时，推荐流程是：

1. 先让当前 agent 把结论更新到 `TASK.md` 或 `PLAN.md`
2. 保证工作树状态可被 `git diff` 清晰表达
3. 新 agent 首先读取 `TASK.md`、共享规则和当前 diff
4. 再继续执行下一步

这能显著降低：

- 对聊天历史的依赖
- 换工具后的重新解释成本
- 不同 agent 之间的信息丢失
### 8.5 标准知识沉淀目录

建议为项目留出轻量的知识沉淀空间，例如：

- `notes/research/`
- `notes/decisions/`
- `notes/experiments/`
- `notes/bugs/`

这类内容不适合全部塞进 README，但也不应全部丢失。

### 8.6 多 agent / subagent 协作模式

现代 AI 编码工具已经普遍不再只是“一个聊天窗口 + 一个模型”。

当前工作流应明确区分四种模式：

#### 模式 A：单 agent 执行模式

适合：

- 简单任务
- 低风险任务
- 明确局部修改

特点：

- 一个主执行面直接完成任务

#### 模式 B：主 agent + 工具内 subagent 模式

适合：

- 单个任务内部需要规划、分析、探索、执行分工
- 使用支持内建 agent 编排的工具

例如：

- `OpenCode` 的 primary agents 和 subagents
- `Claude Code` 的 subagents
- 其他支持内建多 agent 的现代工具

特点：

- 在一个工具生态中完成受控分工
- 适合复杂任务但仍希望统一上下文和权限

#### 模式 C：多 agent 只读并行分析模式

适合：

- 复杂设计选择
- 高风险 bug 分析
- 大上下文阅读
- 需要多模型交叉验证

特点：

- 多个 agent 并行读代码、提方案、做分析
- 由人来收敛结果
- 最终只指定一个执行面动代码

#### 模式 D：多 worktree 独立竞赛模式

适合：

- 方案空间较大
- 需要并行探索
- 希望通过多模型/多路线获得更高质量结果
- 类似 test-time scaling 的任务

特点：

- 多个 agent 在不同 worktree 中各自尝试
- 最终由人比较结果、选优、收敛

本方案建议：

- 允许并行思考
- 允许工具内部并行编排
- 不建议多个外部 agent 无约束同时写同一个 worktree

## 9. 工具分层设计

## 9.1 版本控制层

### 工具

- `Git`
- `git worktree`

### 为什么需要这层

这层负责：

- 版本控制
- 差异审查
- 分支隔离
- 最终合并
- 回滚

### 使用方式

- 主线尽量稳定
- 任务尽量小而明确
- 不同任务不要共用一个工作目录
- 任务完成后进行清理

### 为什么选这些工具

这仍然是软件开发最稳、最通用、最具兼容性的基础设施。即使未来 AI 工具持续变化，Git 层也仍然是最稳定的收敛面。

---

## 9.2 会话与工作台层

### 工具

- `tmux`

### 为什么需要这层

这层负责：

- 远程会话持久化
- 多 pane 协作
- 测试、日志、AI、编辑同时在线
- 长时间任务不中断

### 推荐使用方式

默认可采用两种模型：

- 一个 worktree 对应一个 tmux window
- 一个 worktree 对应一个 tmux session

推荐标准 pane 布局：

- pane 1：`vim`
- pane 2：主 AI CLI 工具
- pane 3：测试或 shell
- pane 4：日志或 `git diff`

### 为什么选 `tmux`

- 远程环境友好
- 无 root 成本低
- 生态成熟
- 已与当前工作流高度匹配

---

## 9.3 阅读与轻编辑层

### 工具

- `Vim`
- `ripgrep`
- `fzf.vim`
- `NERDTree`
- `fd`
- `bat`

### 为什么需要这层

当前工作流中，最常发生的动作不是“深度写代码”，而是：

- 看代码
- 跳转代码
- 搜文件
- 搜字符串
- 改一两处
- 快速验证

### `Vim` 的定位

在本方案中，`Vim` 应定义为：

- 主代码阅读器
- 主导航器
- 主轻编辑器
- 远程环境默认编辑器

它不是“偶尔应急工具”，而是主工作面的一部分。

### 推荐的轻量插件增强

推荐补充：

- `vim-commentary`
- `vim-surround`

原因：

- 高频
- 低维护
- 低心智负担
- 对轻编辑和代码微调非常有价值

### 为什么不主推重型 Vim 生态

并不是说 LSP/Treesitter/DAP 没价值，而是当前主矛盾不在“把 Vim 变成完整 IDE”，而在：

- 多上下文切换
- 阅读效率
- 审阅效率
- worktree 管理
- AI 协作稳定性

---

## 9.4 重编辑层

### 工具

- `VS Code`
- 按需使用 `Cursor` 等 AI IDE

### 为什么需要这层

有些任务依然更适合图形 IDE：

- 大规模多文件修改
- 结构化搜索替换
- 调试
- 较重代码编写
- 需要大量上下文联动时

### `VS Code` 的定位

`VS Code` 不应作为多任务总调度器，而应作为：

- 重编辑工具
- 深度编写工具
- 图形调试工具

它是补充层，而不是唯一主操作面。

---

## 9.5 AI 执行层

### 当前主用工具

- `OpenCode`
- `Gemini CLI`
- `Codex`
- `Claude Code`

### 次级或按需工具

- `GitHub Copilot CLI`
- `Cursor Background Agents`
- `GitHub Copilot coding agent`

### 为什么保留多个 AI 工具

当前没有任何一个 AI 编码工具在以下所有维度都全面领先：

- 开源性
- 免费性
- 终端体验
- 模型质量
- 背景执行能力
- IDE 集成
- 插件/MCP/skills 生态

因此，更合理的做法是：

- 保留一个默认主力本地 agent
- 保留一到两个高质量备选 agent
- 按需使用异步后台 agent
- 在复杂任务中允许多模型并行探索或交叉验证，而不是只按单一路由使用

---

## 10. 各类 AI 编码工具的角色定义

## 10.1 OpenCode

### 建议定位

默认主力本地终端 agent。

### 为什么选它做主力

- 开源
- 免费
- 终端优先
- 支持多 session
- 支持内建 primary agent / subagent 协作
- 支持 MCP
- 与当前终端工作流高度一致
- TUI界面做的很不错

### 最适合做什么

- 日常主力 coding agent
- 多 worktree 并行任务
- 单工具内部的 plan / build / explore 分工
- 文档、测试、重构、局部分析

### 风险

- 和其他 agent 一样，容易生成过量中间文件
- MCP 接太多容易增加上下文噪声

---

## 10.2 Gemini CLI

### 建议定位

高价值的第二本地 agent 和模型路线补充。

### 为什么保留

- 提供模型多样性
- 终端 agent 能力成熟
- 支持 MCP
- 适合做交叉验证和大上下文分析

### 最适合做什么

- 第二意见
- 分析型任务
- 与其他 agent 做对照

---

## 10.3 Codex

### 建议定位

高质量 coding/review 路线。

### 为什么保留

- 在重要实现、深度推理、代码审阅中可作为高质量补充
- 适合在关键任务上做二次验证

### 最适合做什么

- 复杂实现
- 深度审阅
- 较高风险任务的复核

---

## 10.4 Claude Code

### 建议定位

面向长链路、复杂 agent 工作流的强力终端 agent。

### 为什么保留

- 更适合多步骤、长时、工具密集型任务
- 支持内建 subagent 与项目级记忆/指令体系
- 在一些复杂任务上可以作为主力备选

### 最适合做什么

- bug 深挖
- 长流程调研
- 工具链较复杂的任务

---

## 10.5 其他可选参考工具

例如：

- `Aider`
- `Continue`
- `Cline`

这些工具并非无价值，但在当前用户的真实习惯中不属于主工具链。

因此更适合作为：

- 思路参考
- 局部能力借鉴
- 未来按需评估对象

## 10.6 后台异步 agent

### 工具

- `Cursor Background Agents`
- `GitHub Copilot coding agent`

### 建议定位

异步外包层。

### 最适合做什么

- 文档清理
- 小型 bugfix
- 补测试
- 低风险技术债处理

### 为什么不是主力

当前主工作流是：

- 终端优先
- worktree 优先
- 本地可控优先

因此它们更适合作为补充，而不是主骨架。

## 11. MCP、插件、Skills 是否应该建设

答案是：应该，但要克制地建设。

---

## 11.1 总体判断

现代 agent 越来越依赖：

- MCP
- plugins/extensions
- skills
- project rules / custom instructions / memory files

但本方案不建议走“全都装上”的路线，而建议走：

- 少量高价值 MCP
- 稳定、可复用的 skills
- 少量真正高频的插件

目标不是让工具数量最大化，而是让：

- 重复工作标准化
- 上下文更稳定
- 跨 agent 复用更强

这里需要特别强调：

- `OpenCode`
- `Claude Code`
- `Cursor`
- 以及其他现代 AI 编码工具

普遍已经支持不同形式的自定义指令、规则文件、记忆文件或项目级配置。

因此，AI 配置不应被视为“附属小功能”，而应被视为环境设计的一等对象。

---

## 11.2 Skills 的优先级应高于大量 MCP

原因是：

- MCP 解决“agent 能接入什么能力”
- skills 解决“agent 应该如何稳定做事”

对当前工作流来说，后者更重要。

推荐优先建设的 skills：

- `worktree-bootstrap`
- `bug-investigation`
- `test-backfill`
- `research-summary`
- `experiment-variant`
- `docs-polish`

这类 skill 应该编码：

- 任务步骤
- 输出结构
- 验证要求
- 边界条件

---

## 11.3 推荐第一批 MCP 类型

建议第一批只保留少量高价值种类：

- 文档/搜索类 MCP
- Git 或 GitHub 类 MCP
- 文件系统类 MCP
- 数据库类 MCP（仅在确实常用时）

不建议一开始就接一大堆低频 MCP，例如：

- 邮件
- 聊天工具
- 设计工具
- 低频 observability 工具

除非这些已经是日常工作流核心组成部分。

---

## 11.4 为什么要节制

MCP 和插件过多会导致：

- 上下文噪声上升
- agent 乱选工具
- 调试难度增加
- 维护成本上升

本方案更推荐：

- 工具少而稳
- 规则清楚
- 复用性高

---

## 12. 运行时隔离：不仅要隔离代码，还要隔离输出

`worktree` 只解决代码目录隔离，不自动解决运行时隔离。

必须制度化约定这些东西：

- 输出目录
- 日志目录
- 缓存目录
- 端口
- 数据库文件
- 容器名

建议默认规则：

- `outputs/<task>/`
- `logs/<task>/`
- `.cache/<task>/`
- 任务名前缀化的端口和进程命名

这对于以下场景尤其重要：

- 实验
- web 服务
- notebook
- 训练任务
- 长时间后台进程

---

## 13. 项目内建议固定的 AI 相关文件

每个值得长期维护的项目，建议逐步标准化以下文件：

- `AGENTS.md`
- `TASK.md`
- `notes/`
- `PLAN.md`（复杂任务可选）

未来可以考虑再补：

- 项目级 `skills/`
- bug 或 experiment 模板

这些文件不是装饰性的，它们是 AI 协作控制面的组成部分。

## 14. dotfiles 策略

当前仓库可以作为 bootstrap 起点，但中期应该逐步从“单脚本安装”演化为“结构化 dotfiles 系统”。

### 建议的分层

#### 核心层

几乎所有机器共享：

- shell 配置
- git 配置
- tmux 配置
- vim 配置
- 常用 CLI 工具
- worktree helper scripts

#### AI 层

建议单独拆出 AI 相关配置，以便：

- 自己按机器启用或禁用
- 他人单独复用 AI 工作流部分
- 降低非 AI 用户使用 dotfiles 的门槛

建议这一层包括：

- 共享 AI 规则
- 各工具适配规则
- MCP 配置
- skills
- AI wrapper、glue scripts、context scripts
- OpenCode / Claude / Cursor / Gemini / Codex 相关配置

#### 服务器层

面向云服务器和实验室服务器：

- rootless 安装逻辑
- 更轻的工具子集
- 终端优先
- 远程安全默认值

#### 桌面层

面向本地 PC 和实验室电脑：

- `VS Code` 配置
- GUI 相关辅助
- 剪贴板集成
- 可选 IDE 增强

#### 私有层

不进入公开仓库：

- token
- host alias
- 私有环境变量
- 实验室内网配置
- 私有脚本

### 为什么必须分层

因为设备集合已经包含：

- 无 root 服务器
- 云服务器
- 多台 PC
- 实验室电脑

如果继续用一套完全无差异的安装方式，后续一定会越来越脆弱。

如果未来希望公开该仓库让他人使用，这种分层也更合理：

- 非 AI 用户可以只用核心层
- AI 用户可以单独启用 AI 层
- 不同工具偏好的用户可以只挑自己的适配层

---

## 15. 是否应考虑 dotfiles 管理器

本方案不强制立刻迁移，但中期建议认真考虑：

- `chezmoi`
- `yadm`
- 或者至少演进成更结构化的自定义 bootstrap 系统

### 为什么值得考虑

- 更好支持机器差异
- 更好支持模板
- 更好处理 secret
- 更方便公开/private 分层
- 更适合设备数量较多的长期维护

## 16. 建议补充的通用命令与工具

这些工具不是“流行就装”，而是因为它们解决了高频摩擦。

### 16.1 `zoxide`

#### 作用

基于历史记录进行目录跳转。

#### 为什么需要

多 worktree、多项目环境下，切目录是高频动作。

#### 为什么选它

相比纯 alias，它更适合动态目录集合。

---

### 16.2 `fd`

#### 作用

快速找文件。

#### 为什么需要

配合 `fzf` 能显著降低文件导航成本。

#### 为什么选它

比原生 `find` 更适合日常交互式使用。

---

### 16.3 `bat`

#### 作用

更适合阅读代码和配置文件。

#### 为什么需要

当前工作流非常偏“读代码”和“看配置”。

#### 为什么选它

提升阅读体验，但不会增加太多复杂度。

---

### 16.4 `delta`

#### 作用

增强 diff 可读性。

#### 为什么需要

AI 时代人的核心瓶颈之一是 review。

#### 为什么选它

它直接作用于最重要的人工验证面：`git diff`

---

### 16.5 `jq`

#### 作用

查看和处理 JSON 输出。

#### 为什么需要

现代工具、日志、接口和实验输出中 JSON 很常见。

#### 为什么选它

轻量、通用、高价值。

---

### 16.6 `atuin`

#### 作用

增强 shell 历史搜索，可选跨设备同步。

#### 为什么需要

当前是多设备、多服务器环境，而且 shell 命令复用频繁。

#### 为什么选它

相比原生 shell history，检索能力更强，更适合长期积累。

---

## 17. 建议提供的命令接口

建议通过 shell function 或脚本提供一组统一命令，而不是每次手打原生命令。

建议命令包括：

- `wt-new <type> <name>`
- `wt-list`
- `wt-enter <name>`
- `wt-rm <name>`
- `wt-prune`
- `wt-context`
- `task-handoff`
- `task-init`
- `task-status`
- `task-clean`

这些命令的意义在于：

- 降低操作摩擦
- 降低手误
- 把方法论固化为工具

其中比较关键的两个命令是：

- `wt-context`
  用于打包当前 worktree 的任务说明、共享规则、当前 diff 和目录摘要，便于切换到另一个 AI 工具继续工作
- `task-handoff`
  用于生成结构化接力说明，明确当前进度、下一步、风险和待验证项

## 19. 风险与失败模式

### 19.1 worktree 泛滥

太多过期 worktree 会导致混乱。

缓解方式：

- 定义生命周期
- 定期清理
- 合并或废弃后及时 remove 和 prune

### 19.2 AI 产物泛滥

AI 很容易生成：

- 临时脚本
- 草稿文档
- 分析文件
- 日志
- 重复副本

缓解方式：

- 任务级临时目录
- 本地 ignore 规则
- 明确清理习惯

### 19.3 运行时冲突

代码目录隔离后，仍可能在这些层面冲突：

- 端口
- 缓存
- 数据库
- 容器名

缓解方式：

- 任务名前缀资源
- 默认输出/日志/缓存规范

### 19.4 工具过度扩张

AI 工具、MCP、IDE、插件太多会降低清晰度。

缓解方式：

- 每层有一个默认主力
- 其他工具做备选，不要每个任务全开

## 20. 近期优先级建议

### 优先级 1

- 正式引入 `AGENTS.md`
- 正式引入 `TASK.md`
- 补 `vim-commentary`
- 补 `vim-surround`
- 增加 `zoxide`、`fd`、`bat`、`delta`、`jq`
- 增加 worktree helper commands

### 优先级 2

- 标准化 tmux task layout
- 先做少量高价值 skills
- 先接少量高价值 MCP
- 优化 shell prompt 和 shell history

### 优先级 3

- 重构 dotfiles 仓库结构
- 拆分 core/server/desktop/private 层
- 评估 `chezmoi` 或 `yadm`

## 21. 最终建议

本方案建议将整个系统建设为一套：

- 终端优先
- AI 原生
- 多 worktree
- 多设备
- 强调 review
- 注重可迁移性

的个人开发系统。

核心建议不是“装更多工具”，而是建立一个稳定的分层架构：

- Git 管理真相
- worktree 管理任务隔离
- tmux 管理活动上下文
- Vim 管理高频阅读与轻编辑
- VS Code 处理重编辑
- AI agent 作为可替换执行器
- skills 和规则文件稳定 AI 行为
- dotfiles 负责多环境一致性

这套架构比围绕某个单一 IDE 或单一 AI 产品构建环境更稳，也更适合：

- 远程开发
- 科研实验
- 多任务并行
- AI 协作
- 多机器长期维护
